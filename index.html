LESSON 1 / 5~20 min

Understanding Queues: Definition and Core Properties

Why the fairest data structure in computer science is also one of the simplest: first come, first served.

Why queues matter

Imagine waiting in line at a coffee shop, each person patiently moving forward as the barista serves the next customer. That familiar scenario is a perfect model for a queue — a data structure that is just as important in computer science as it is in daily life. Queues help computers manage tasks, requests, and data in an orderly fashion, ensuring fairness and efficiency, from print spooling to web server request handling.

Structure — what is a queue?

A queue is a linear data structure that restricts how items are added and removed. Items can only be added at one end, called the back, and removed from the other end, called the front. This guarantees First-In, First-Out (FIFO): the first item added is always the first one removed — no item can skip ahead of one that arrived earlier.

Key queue terminology

TermDefinition
QueueA linear data structure where items are added at the back and removed from the front, following the FIFO principle.

FIFOFirst-In, First-Out — the first item added to a queue is the first one to be removed.
FrontThe end of the queue from which items are removed (dequeued).
BackThe end of the queue at which items are added (enqueued).
EnqueueThe operation of adding an item to the back of the queue.
DequeueThe operation of removing and returning the front item from the queue.

Operation — enqueue and dequeue

Enqueue: add an item to the back. Starting empty, enqueue 'A' — A sits alone. Enqueue 'B' next and B joins behind A, at the back; A is still at the front. Dequeue: remove the front item. With A at the front and B at the back, dequeue removes and returns A — B now becomes the new front. Notice enqueue and dequeue act on opposite ends — that's what makes a queue different from a stack.

Pitfall — front and back are not interchangeable

A common mix-up: treating a queue like it has one flexible end you can add to or remove from at will. It doesn't. Enqueue always happens at the back; dequeue always happens at the front. Enqueueing at the front, or dequeueing from the back, isn't a shortcut — it silently breaks the FIFO guarantee the whole structure exists to provide.

Think of a queue like a line at a ticket counter: new arrivals join the back, and only the person who has been waiting longest — at the front — gets served next. A printer works the same way: the first document submitted is the first one printed, however many jobs pile up behind it. Web servers use the same idea to handle incoming requests fairly, and breadth-first search uses a queue to explore a graph level by level, in the exact order nodes are discovered.

First-In, First-Out (FIFO) — not LIFO, not random access, and not priority-based removal. The item that has waited longest always leaves next.
A queue contains A, B, and C, with A at the front (the first one enqueued). What happens when you call dequeue()?

LESSON 2 / 5~20 min

Comparing Stacks and Queues: Key Differences

Same family of linear structures, opposite rules — knowing which one to reach for is half the battle.

Two linear structures, two disciplines

Stacks and queues are both linear data structures: their elements are arranged in a sequence. But they differ in how items are added and removed. A stack allows insertion and removal at only one end (the top). A queue insists on two different ends — items are added at the back and removed from the front. That single design decision is what gives each structure its entire personality.

Structure — key terms, side by side

Stack: a linear data structure where items are added and removed from the same end, called the top — LIFO (Last-In, First-Out). Queue: a linear data structure where items are added at the back and removed from the front — FIFO (First-In, First-Out).

In the stack, D (the last item pushed) is the first one popped. In the queue, A (the first item enqueued) is the first one dequeued — even though D just joined at the back.

Operation — adding, removing, and processing order

Stacks add with push and remove with pop — both at the top. Queues add with enqueue (back) and remove with dequeue (front). The knock-on effect: a stack always processes the most recently added item next; a queue always processes the longest-waiting item next.

Cost — picking the structure that matches the problem

Choosing between a stack and a queue isn't about performance first — it's about which order the problem actually needs. Need to reverse or undo the most recent action? That's LIFO — reach for a stack. Need to guarantee fairness, so the earliest request is always handled first? That's FIFO — reach for a queue. Using the wrong one doesn't just cost you speed; it gives you the wrong order entirely.

Pitfall — don't assume they're interchangeable

Because stacks and queues look similar (linear, restricted access), it's tempting to assume they behave similarly too. They don't: neither allows random access to elements in the middle, but a stack's single end means push and pop compete for the same spot, while a queue's two ends mean enqueue and dequeue never touch the same spot at all.

LESSON 3 / 5~15 min

Abstract Data Types: The Queue ADT

The same trick you saw with the Stack ADT, applied to a different set of rules: define the what before you touch the how.

Abstraction, again

An Abstract Data Type (ADT) is a model that defines a data structure by the operations it supports, without pinning down how those operations are implemented. This separation lets programmers use a data structure based on its behaviour, not its internal machinery. The Queue ADT applies exactly that idea to the enqueue/dequeue/FIFO rules from Lessons 1–2.

Structure — key terms

ADT: a model that defines a data structure by its allowed operations and behaviours, not by its implementation details. Queue ADT: an ADT where items are added at the back and removed from the front, following FIFO. Interface: the set of operations an ADT provides (enqueue, dequeue, isEmpty, …). Implementation: the actual code that carries those operations out internally.

The Queue ADT interface

OperationWhat it does
Queue()Creates a new, empty queue.
isEmpty()Returns a boolean indicating whether the queue is empty.
__len__()Returns the number of items currently in the queue.
enqueue(item)Adds the given item to the back of the queue.
dequeue()Removes and returns the front item — cannot be called on an empty queue.
front() / peek()Returns the front item of a non-empty queue without removing it.

Operation — why abstraction matters

Hides complexity: you can use a queue without knowing whether it's built on a list, an array, or linked nodes. Enables flexibility: the same ADT can be implemented multiple ways — swap one out for another without changing how client code calls it. Supports reusability: once the interface is defined, it can be reused across many programs regardless of the underlying implementation.

Same interface, different implementations

Whether the queue is built on an array, a Python list, or linked nodes, the user interacts with it only through enqueue, dequeue, isEmpty, and __len__ — the implementation stays hidden behind that interface.

ApproachStrengthsWeaknesses
Python listSimple and quick to write; built-in methods handle the heavy lifting.Removing from the front shifts every remaining element — inefficient for large queues.
Circular arrayNo shifting needed — enqueue and dequeue both become O(1); predictable, fixed memory use.Fixed capacity; resizing beyond the initial size adds complexity.
Linked listDynamic sizing; enqueue and dequeue stay efficient without shifting elements.Extra memory for pointers; more complex to implement correctly.

Cost — the trade-off in one line

A Python list wins on simplicity. Circular arrays and linked lists win on performance, because both avoid the shifting cost that comes from removing an item from the front of a plain list — a cost Lesson 4 will make concrete, and Lesson 5 will solve.

Pitfall — don't assume every implementation behaves the same underneath

Because the Queue ADT interface always looks the same, it's tempting to assume every implementation performs the same too. It doesn't — a list-based dequeue() and a circular-array-based dequeue() both "remove and return the front item," but one shifts every remaining element and the other doesn't. Same interface, very different cost.

Quick self-check

Which of the following best describes an Abstract Data Type (ADT)?
A set of operations and behaviours defined independently of implementation — not a specific memory layout, a language feature, or a sorting algorithm.
True or false: because the Queue ADT interface stays the same, you can swap a list-based queue for a circular-array-based queue without changing how client code calls it.
True — that's the whole point of an ADT. Client code only ever calls enqueue, dequeue, isEmpty, and __len__, regardless of what's happening underneath.
LESSON 4 / 5~25 min

Implementing the Queue ADT in Python

From blueprint to running code — and the first hint that not every O(1)-looking operation stays that way.

From concept to code

The Queue ADT defines what a queue should do, but not how it should be built. Python's built-in list gives a fast, familiar way to get a working queue on the page: append() to add at the back, and a removal from index 0 to take from the front.

Python implementation

# Implementation of the Queue ADT using a Python list.
class Queue:
    # Creates an empty queue.
    def __init__(self):
        self._qList = list()

    # Returns True if the queue is empty.
    def isEmpty(self):
        return len(self._qList) == 0

    # Returns the number of items in the queue.
    def __len__(self):
        return len(self._qList)

    # Adds the given item to the queue.
    def enqueue(self, item):
        self._qList.append(item)

    # Removes and returns the first item in the queue.
    def dequeue(self):
        assert not self.isEmpty(), "Cannot dequeue from an empty queue."
        return self._qList.pop(0)
MethodWhat it does
enqueue(item)Appends item to the end of the internal list — it becomes the new back.
dequeue()Asserts the queue isn't empty, then removes and returns self._qList[0] via .pop(0).
isEmpty()Returns True when len(self._qList) == 0.
__len__()Returns len(self._qList) — lets you call len(queue) directly.

Structure — front = index 0, back = last index

The list's own order stands in for the queue's order: index 0 is the logical front, and the last index is the logical back. Enqueue appends to the end; dequeue removes from index 0.

Cost — enqueue is O(1), dequeue is O(n)

Appending to the end of a Python list is O(1) — no shifting needed. But pop(0) removes from the front, which forces every remaining element to shift one position to the left, making dequeue O(n). This is the opposite trade-off from a stack, where both push and pop touch the same end (the end of the list) and stay O(1). A queue's two ends are precisely what breaks that symmetry with a plain list.

Pitfall — assuming dequeue is O(1) just because pop() is

Python's list.pop() with no argument removes from the end in O(1). list.pop(0) removes from the front and is O(n), because every remaining item has to move. It's easy to see "pop" and assume constant time — the argument is what changes everything.

Looking ahead

For small or simple applications, list-based dequeue's O(n) cost rarely matters. But for large queues or performance-critical code, Python's collections.deque (optimised for fast appends and pops at both ends) is a common quick fix — and Lesson 5 shows the classic computer-science solution: rebuilding the queue on a circular array so nothing ever has to shift.

Quick self-check

Which Python list method is typically used to perform the enqueue operation in a list-based queue?
append() — it adds the new item to the end of the list, which represents the back of the queue.
Which Python list method is typically used to dequeue the front item from a list-based queue?
pop(0) — it removes and returns the item at index 0, which represents the front of the queue.
True or false: using a Python list, both enqueue and dequeue operations always take constant (O(1)) time.
False. Enqueue (append) is O(1), but dequeue (pop(0)) is O(n) because it must shift every remaining element forward.
LESSON 5 / 5~25 min

Implementing Queues with Circular Arrays

Turn the array into a loop, and the shifting problem from Lesson 4 disappears entirely.

Why circular?

A circular array treats its two ends as connected, forming a logical circle instead of a straight line. When you reach the end of the array, you simply "wrap around" back to the beginning. For a queue, this means enqueue and dequeue can reuse freed-up space at the front without ever shifting the elements in between — solving exactly the O(n) dequeue problem from Lesson 4.

Structure — key terms

Circular array: an array that connects its end to its beginning, letting elements wrap around for efficient space use. Fixed-size buffer: a storage area with a set capacity. Front pointer: the index of the item next in line to be dequeued. Back pointer: the index where the next enqueued item will go. Wrap-around: moving from the end of the array back to index 0 to keep reusing the buffer.

The four building blocks

  1. Initialize — create an array of fixed size; set front and back pointers, usually both starting at 0; set a count variable to 0.
  2. Enqueue — insert the item at the position after back; advance back = (back + 1) % array_size; increment count.
  3. Dequeue — return the value at front; advance front = (front + 1) % array_size; decrement count.
  4. Check full / empty — before enqueueing, check whether count == array_size (full); before dequeueing, check whether count == 0 (empty).

Operation — advancing pointers with modulo arithmetic

Pointers wrap using the modulo operator. If the array size is 5 and back is at index 4, advancing it means back = (4 + 1) % 5 = 0 — it wraps to index 0 instead of running off the end. This keeps every pointer permanently inside the bounds of the array, no matter how many enqueues and dequeues have happened.

Cost — O(1) enqueue and dequeue, no shifting, ever

Because both pointers simply advance (wrapping with modulo when needed) instead of shifting elements, enqueue and dequeue are both O(1) — solving the exact dequeue bottleneck the Python-list queue had in Lesson 4.

Pitfall — telling full and empty apart

A subtle trap: both a full queue and an empty queue can end up with front and back pointing at the same index — pointer position alone is ambiguous. The standard fix is a count variable: count == 0 means empty, count == array_size means full. (Some implementations instead reserve one slot that's always left empty.) Skipping this check risks silently overwriting unread data or reading garbage from an empty queue.

Quick self-check

In a circular array queue, what happens when the back pointer advances past the last array index?
It wraps around to index 0 — that's the entire point of the modulo arithmetic (back = (back + 1) % array_size).
What is the primary advantage of using a circular array for implementing a queue?
It eliminates the need for shifting elements during dequeue operations — the front and back pointers simply advance instead.
How do circular array queues typically distinguish a full queue from an empty one?
With a count variable — count == 0 means empty, count == array_size means full — since front and back can coincide in both cases.
CRAM SHEET~8 min — last pass before practice

One-Page Recap

If you only remember these, remember these.

1. What a queue is

A linear structure where items are added at the back (enqueue) and removed from the front (dequeue), following FIFO (First-In, First-Out). Two ends, not one — that's the whole difference from a stack.

2. The Queue ADT

enqueue(item) — add to back. dequeue() — remove return front. isEmpty() / __len__() — report state. front()/peek() — view the front without removing it. The ADT defines these operations; it says nothing about how they're implemented.

3. Python list implementation its cost

enqueue = append() → O(1). dequeue = pop(0) → O(n), because every remaining item shifts left. This asymmetry is the whole reason circular arrays exist.

4. Stacks vs. queues, in one line

Stack: one end (top), push/pop, LIFO — good for reversing/undoing. Queue: two ends (front/back), enqueue/dequeue, FIFO — good for fairness/order preservation.

5. Circular arrays: the fix

Treat the array's two ends as connected. Advance front/back with modulo arithmetic ((i + 1) % size) so they wrap instead of running off the array. Both enqueue and dequeue become O(1) — no shifting, ever.

6. Full vs. empty

front and back can land on the same index in both states — pointers alone can't tell them apart. Track a count variable: count == 0 → empty, count == array_size → full.
